# v15：expert 缩减的真实 perplexity 代价 + 最终权衡曲线

**日期**：2026-07-15
**执行**：GPU 4，HF transformers（patch Qwen3MoeSparseMoeBlock.top_k）
**目的**：把 v14 的"权重改动"代理指标换成**真实 perplexity**——回答用户 idea"省搬运 vs 掉精度"到底值不值。
**脚本**：`scripts/run_v15_perplexity.py`

---

## 方法
干预 = **top-k 缩减**：每个 token 只用最强的 r 个专家（而非 8）。这是干净、well-defined 的干预，且是聚集代价的**保守上界**（直接丢掉最弱专家，至少和"劝导到相似的已激活专家"一样有害）。patch 全部 48 个 MoE block 的 `top_k`，在 agent 风格文本上测 perplexity。

---

## 结果：真实 PPL vs 每 token 用的专家数
| 每 token 专家数 | 丢弃最弱 | **真实 PPL 增加** |
|---|---|---|
| 8（baseline） | 0 | 0%（PPL 5.769） |
| 7 | 1 | **+2.2%** |
| 6 | 2 | **+5.5%** |
| 5 | 3 | **+7.6%** |
| 4 | 4 | +23.0% |
| 3 | 5 | +53.0% |
| 2 | 6 | +261% |

**明显的拐点在 top4–5 之间**：丢最弱 1–3 个专家（top7/6/5）精度代价温和（2–8%）；一旦砍到 top4 以下，精度崩塌。

---

## ★ 最终权衡曲线（合并 v14b 搬运节省 + v15 真实精度）

把"丢最弱 k 个专家"对应到 v14b 的搬运节省（真实 agent decode batch 6–20，即 batch=8 附近）：

| 干预 | 真实 PPL 增加（v15） | 搬运节省（v14b, batch≈8） |
|---|---|---|
| 丢最弱 1 个（top7） | +2.2% | ~4% |
| **丢最弱 2 个（top6）** | **+5.5%** | ~13% |
| **丢最弱 3 个（top5→用 rank≥6 聚集口径）** | **+7.6%** | **~17–20%** |
| 丢最弱 4 个（top4） | +23%（崩） | ~25% |

**甜蜜点**：**丢最弱 2–3 个专家 → 搬运省 13–20%，PPL 只涨 5.5–7.6%。**

> 重要：这是**保守上界**。v15 是"直接丢弃"最弱专家；用户 idea 是"**劝导到已激活的相似专家**"（那个专家仍会计算并贡献），实际精度损失应**小于**这个上界。而且 v13 已测得被劝导的 rank6-8 专家 router 置信度低、权重仅 7%——真实精度损失预计明显 < 7.6%。

---

## 结论：idea 值不值？—— 有条件地值，且方向明确

1. **正向**：在真实 low-concurrency agent decode（batch 6–20）上，**省 ~17–20% 专家搬运，真实 PPL 上界仅 +5.5–7.6%**（实际更低）。搬运是 decode 的绝对瓶颈（103:1），省 20% 搬运 ≈ decode 提速接近 20%（memory-bound 下时间∝搬运）。
2. **拐点清晰**：不能贪心——丢超过 3 个专家（top4 以下）精度崩塌。安全区是"动最弱 2–3 个"。
3. **与已有工作的关系**：这条 top-k 缩减曲线本身对应 "dynamic-k" 工作；用户的**批内聚集**独特点在于"劝导到已激活专家"（省搬运的同时保留计算贡献）→ 精度应优于纯缩减。v15 给了它一个**保守的精度上界背书**。

---

## 局限（诚实）
1. **top-k 缩减 ≠ 完整的批内聚集**：v15 是保守上界；真正的"劝导到活跃专家"需要实现 batch 级重路由 + 前向，才能测到真实（更低）的精度代价。这是下一步。
2. perplexity 用 3 段 agent 文本；扩到标准 benchmark（下游任务 pass@k）更严谨。
3. **搬运节省是理论值**（激活专家数下降）；真实加速要落到 kernel（利用减少的专家数），未实测端到端 latency。
4. 是否均匀丢最弱 vs 自适应（简单 token 多丢、难 token 不丢）未探索——自适应应更优（呼应 MS "fragile tokens" 发现）。
